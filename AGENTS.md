# Repository Guidelines

This file is the shared source of truth for AI coding assistants (Claude Code, Codex, Aider, etc.) working in this repo. `CLAUDE.md` is a stub that points here.

## Workspace layout

Cargo workspace of five crates plus a TypeScript/React frontend. See `Cargo.toml` for members.

- `putty_core/` — transport-agnostic `Connection` trait and `ConnectionManager`. Transport backends (`serial`, `ssh`) are Cargo features.
- `putty_storage/` — JSON profile store (`$XDG_CONFIG_HOME/putty_rs/profiles/`) with OS keyring for SSH secrets.
- `putty_cli/` — `putty-rs` binary. Features `serial`, `ssh`, `storage` all default-on and conditionally compile CLI subcommands.
- `putty_grpc_server/` — `tonic` gRPC + gRPC-Web service exposing `ConnectionManager` and `ProfileStore`. Proto lives in `putty_grpc_server/proto/putty_interface.proto`; `build.rs` generates both server stubs and types via `tonic-build`.
- `putty_rs_web/` — single-binary launcher that spawns `putty_grpc_server::run()` on :50051 and an `axum` static server on :8080 serving `webui/dist` (embedded via `rust-embed` in release builds). Opens a browser on start.
- `webui/` — React 19 + Vite + xterm.js frontend. Talks to the backend via Connect-Web; proto stubs live in `webui/src/generated/` and are regenerated by `webui/scripts/gen_proto.py`.
- `python_grpc_client/` — reference Python CLI that speaks the same proto.

Architecture and runtime diagrams live under `docs/`.

## Common commands

Build and test from the repo root:

```bash
cargo build --workspace
cargo test --all                                               # excludes hw-tests
cargo test -p putty_core --features hw-tests -- --nocapture    # all hardware tests
cargo test -p putty_core --test hw_ssh --features hw-tests -- --nocapture   # single test file
cargo fmt --all
cargo clippy --all-targets --all-features -- -D warnings
```

Feature-gated CLI builds (mirror the install variants in `putty_cli/README.md`):

```bash
cargo build -p putty-rs                                                             # default: serial+ssh+storage
cargo build --manifest-path putty_cli/Cargo.toml --no-default-features --features serial
cargo build --manifest-path putty_cli/Cargo.toml --no-default-features --features ssh,storage
cargo run -p putty-rs -- --help
```

Run the servers / frontend:

```bash
cargo run -p putty_grpc_server          # gRPC-Web on 0.0.0.0:50051
cargo run -p putty_rs_web               # gRPC on :50051 + static UI on :8080 (opens browser)
cd webui && npm ci && npm run proto && npm run dev   # Vite dev server on :5173 (proxies /rpc/* to backend)
cd webui && npm run build && npm run lint
```

System deps on Ubuntu: `protobuf-compiler` for `tonic-build`. No native TLS/SSL libraries are needed — SSH is handled by the pure-Rust `russh` crate.

## Architecture

**Core I/O model** (`putty_core/src/core/connection_manager.rs`): `ConnectionManager` is the single entry point used by every frontend (CLI, gRPC, Python, web). It is `Clone` (internally `Arc<Mutex<HashMap<id, ConnectionIOHandle>>>`) and cheap to share.

`add_connection(id, Box<dyn Connection>)` takes ownership of a not-yet-connected transport, calls `connect()`, then spawns one Tokio task per connection with two channels:

- `mpsc::Sender<IoEvent>` for `write_bytes` / `stop_connection` — frontend → I/O task.
- `broadcast::Sender<Vec<u8>>` for `subscribe` — I/O task → any number of listeners. Multiple UIs (e.g. CLI + web) can subscribe to the same connection.

Inside the task, `tokio::select!` multiplexes the write/stop mpsc against `conn.read()`. This is the only place where transports are driven; frontends never touch `Connection` directly after registering.

**Transports** (`putty_core/src/connections/`) implement the `async_trait Connection`: `connect`, `disconnect`, `read`, `write`. `SerialConnection` wraps `tokio_serial::SerialStream`. `SshConnection` uses the pure-Rust async `russh` crate: `connect` performs the handshake + auth (password or key, via `SshConnection::with_key`) and opens a shell channel with a PTY; `write` calls `Channel::data`; `read` drives `Channel::wait` and copies `ChannelMsg::Data` / `ExtendedData` into the caller's buffer, stashing any overflow as leftovers for the next read. Both transports are behind the `serial` / `ssh` Cargo features, so `putty_core` consumers must opt in; `cfg(feature = "…")` gates propagate up through `putty_cli`.

**Profile storage** (`putty_storage`): Serial profiles are plain JSON. SSH profiles write a redacted JSON and store the password in the OS keyring under service `putty_rs`, user `putty_rs:<profile-name>`. `ProfileStore::list()` lazily rehydrates the password from the keyring on read.

**gRPC server** (`putty_grpc_server/src/server.rs`): `ConnectionService` owns one `ConnectionManager` and one `ProfileStore` for the whole process. `create_remote_connection` generates a UUID, constructs the transport (or resolves it from a stored profile), registers it, and returns the id. `read` returns a server-streaming `ReceiverStream` that forwards `broadcast` chunks to the client. The server enables `accept_http1` + `GrpcWebLayer` + permissive CORS so browsers can call it directly.

**Frontend**: `webui/src/components/useGrpc.ts` holds the Connect-Web transport; `App.tsx` + `TerminalPane.tsx` wire keystrokes to `Write` RPCs and the `Read` stream into xterm.js.

### Regenerating proto

After editing `putty_grpc_server/proto/putty_interface.proto`:

- Rust side regenerates automatically via `putty_grpc_server/build.rs` on next `cargo build`.
- Frontend: `cd webui && npm run proto` (also runs as `prebuild`).
- Python: the one-time setup in `README.md` using `grpc_tools.protoc` + `protoletariat`.

## Coding style & naming conventions

Rust uses the standard `rustfmt` style with 4-space indentation, `snake_case` for modules/functions, and `CamelCase` for types. Keep async boundaries explicit and prefer small modules below `putty_core/src/` grouped by domain (`connections/`, `storage/`, `utils/`). TypeScript in `webui/src/` follows ESLint defaults, 2-space indentation, and `PascalCase` component filenames such as `ProfilesModal.tsx`.

## Testing

Integration tests live under `putty_core/tests/`, named for the behavior they cover (e.g. `profile_store.rs`, `roundtrip_and_write.rs`). Deterministic tests use `FakeConnection` and helpers in `putty_core/tests/common/`.

Tests whose files/items are `#![cfg(feature = "hw-tests")]` spin up real infrastructure and only build with that feature:

- `hw_serial.rs` is also `target_os = "linux"` — it uses `socat` to create a pty pair.
- `hw_ssh.rs` is `cfg(unix)` — it spawns a throwaway `sshd` with a generated keypair.

If adding a transport-level integration test that needs external binaries, gate it behind `hw-tests` and reuse the helpers in `tests/common/`.

## Releasing

Automated via `release-plz` (see `RELEASING.md` and `release-plz.toml`). Only `putty_core`, `putty_storage`, and `putty-rs` are managed/published. Version bumps come from Conventional Commit messages on `main` (`fix:` → patch, `feat:` → minor, `type!:` → major). Cross-platform binary archives for `putty-rs` are attached by `.github/workflows/release-artifacts.yml` on each GitHub Release. `putty_grpc_server` and `putty_rs_web` are not published to crates.io.

## Commit & pull request guidelines

Recent history favors short, imperative commit subjects. Keep commit titles concise, present tense, focused on one change, and Conventional-Commit-formatted when they affect release versioning (`fix:`, `feat:`, `type!:`). Pull requests should summarize the user-visible impact, note any cross-crate or proto changes, list verification commands run, and include screenshots only when `webui/` behavior changes.
